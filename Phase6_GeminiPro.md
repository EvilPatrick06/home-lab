# Phase 6: AI DM & Discord Bot System Research Report

## 1. AI DM Context Awareness & Features

**Is the AI DM missing any features, behaviors, or context awareness?**
The AI DM is highly sophisticated and possesses a robust context-building pipeline (`src/main/ai/context-builder.ts`). It aggregates:
- **Rulebook RAG:** Excerpts from the 2024 PHB/DMG.
- **SRD Data:** JSON lookups for spells, items, and monsters.
- **Character Data:** Full character sheets for the acting character, abbreviated sheets for the rest of the party.
- **Campaign Data & Memory:** Persistent world state, NPC relationships, and session logs via `MemoryManager`.
- **Game State:** Active map creatures, fog of war, and time of day.

**Missing/Partially Implemented Features:**
- **Acting Character Wiring:** The `context-builder.ts` supports an `actingCharacterId` parameter to provide a full sheet for the active player and abbreviated sheets for others. However, this parameter is never wired through `src/main/ai/types.ts` or `src/main/ai/ai-service.ts` (as noted in `__Planning_Consolidated.md`).
- **Stat Mutation Approval:** Stat mutations (damage, healing, conditions) generated by the AI DM bypass user approval in `src/renderer/src/hooks/use-game-effects.ts`, which can lead to untrustworthy automated state changes.
- **Silent Failures:** SRD data loading errors are silently swallowed in `context-builder.ts:162-163`.
- **Packaged Build Pathing:** `loadMonsterData()` in `context-builder.ts` uses `__dirname` to find renderer public data, which is highly likely to break in packaged Electron builds.

## 2. Prompt Structure and Quality

**Are prompts well-structured and producing quality, rule-accurate responses?**
The main prompt (`src/main/ai/dm-system-prompt.ts`) is incredibly comprehensive (~800 lines). It enforces:
- **Narrative Voice:** Strict rules against markdown headers/bullets in narration, enforcement of second-person (solo) or third-person (multiplayer) prose, and sensory details.
- **Rule Accuracy:** Explicit citations of 2024 PHB rules (Concentration, Exhaustion, Unarmed Strikes, Falling, etc.).
- **JSON Formatting:** Strict schemas for `[STAT_CHANGES]` and `[DM_ACTIONS]`.

**Structural Issues:**
- **Monolithic Design:** The prompt is a single massive string. While there are modular additions (`DM_TOOLBOX_CONTEXT`, `PLANAR_RULES_CONTEXT`), the core prompt lacks modular loading based on campaign setting or context, consuming a massive token budget on every request.
- **Type Mismatches:** The prompt instructs the AI to use the `place_creature` DM action (which auto-populates stats from SRD), but `place_creature` is missing from the `DmAction` TypeScript union in `src/main/ai/dm-actions.ts`.

## 3. Discord Bot Integration

**Is the Discord Bot fully integrated and all commands working?**
The project features a dual-layered Discord integration:
1. **VTT Discord Bridge (`src/main/discord-integration/discord-service.ts`):** Forwards AI DM narrative text from the Electron VTT to a Discord channel via Webhook or Bot API. It successfully strips out VTT-specific metadata (`[DM_ACTIONS]`, `[STAT_CHANGES]`) so Discord users only see the story.
2. **Standalone Pi Bots (`BMO-setup/pi/discord_dm_bot.py` & `discord_bot.py`):** Python-based bots that handle voice channels, TTS (via Fish Audio), and STT (via Groq). 

**Commands:**
The standalone `discord_dm_bot.py` implements a rich set of slash commands:
- Session Management: `/dm start`, `/dm stop`, `/dm status`
- Mechanics: `/roll`, `/initiative`, `/recap`
- Lookups: `/spell`, `/item`, `/condition`, `/monster`
- Generators: `/npc`, `/encounter`, `/tavern`, `/loot`

**Is the bot correctly bridging Discord voice/text channels to the VTT session?**
The bridging is functional but asymmetric. The VTT can push text to Discord (`sendNarrationToDiscord`), and the VTT can trigger the Pi to start/stop the DM bot via HTTP (`src/main/bmo-bridge.ts`). However, the standalone Python bot maintains its own isolated `DMSession` state, combat log, and RAG memory. Actions taken purely in the Discord voice channel (via the Python bot) do not automatically sync token positions or stat changes back to the Electron VTT game board.

## 4. AI DM Capabilities

**Is the AI DM able to run combat, narrate exploration, and handle social encounters?**
Yes, the system is explicitly designed for all three pillars of D&D:
- **Combat:** The AI tracks HP, AC, and conditions. It can roll initiative (`start_initiative`), apply area effects (`apply_area_effect`), and use mob attack rules.
- **Exploration:** The AI manages travel pace (`set_travel_pace`), advances in-game time (`advance_time`), controls ambient lighting/fog of war, and dynamically switches audio tracks (`play_ambient`).
- **Social:** The AI tracks NPC attitudes (Friendly/Indifferent/Hostile), logs relationship changes (`set_npc_relationship`), and handles the 2024 Influence Action mechanics.

## 5. Summary of Missing, Broken, or Partially Implemented Items

1. **`place_creature` Action:** Documented in the prompt but missing from the TypeScript `DmAction` union.
2. **State Change Trust:** AI-generated `[STAT_CHANGES]` are applied automatically without DM/Player approval routing.
3. **`actingCharacterId` Context:** The context builder supports focusing the context window on the currently acting character to save tokens, but the upstream AI service does not pass this ID down.
4. **Discord/VTT State Sync:** The Discord Python bot and the Electron VTT maintain separate session states. Discord players get the narrative, but their mechanical state isn't deeply synchronized with the VTT's visual grid.
5. **Packaged Build Paths:** `loadMonsterData()` uses `__dirname` which will fail in production Electron builds.
6. **Error Handling:** SRD loading errors in the context builder are silently caught and ignored, potentially leaving the AI without critical spell/monster data.
